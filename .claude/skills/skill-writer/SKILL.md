# Skill-Writer — Blueprint → Candidate Skill

Reads a blueprint produced by `/discover-execute`, extracts the load-bearing knowledge (ADRs, Recommendations, key citations to `.claude/knowledge-base/references/`), and writes a candidate SKILL.md that can later be promoted to a first-class skill consumed by `/to-plan`.

## Cycle contract

This skill is **phase 6** (skill-distillation tail) of [`cycle-discover`](../../rules/cycle-discover.md). The cycle rule is the **source of truth** for the full chain (upstream `/discover-*` skills + this skill + `/skill-validator` + `/skill-register`), gates, anti-patterns, stop conditions, and rollback.

**Read `cycle-discover.md` before invoking this skill.** This SKILL.md retains only phase-specific detail.

## When to Trigger

User invokes `/skill-writer {blueprint-slug}` explicitly after `/discover-confidence` (or `/discover-improve` followed by re-score) returns verdict ≥ SHIPPABLE_WITH_CAVEATS. The skill-distillation tail (`/skill-writer → /skill-validator → /skill-register`) is **not orchestrated automatically today** — each phase is human-invoked. The chain in `cycle-discover.md` documents the recommended order.

Do NOT trigger when blueprint verdict is INVALID or NON_SHIPPABLE — the blueprint is structurally broken; generating a skill from it would propagate the defect.

## Output

Single candidate skill at:

```
.claude/skills/generated/{topic}-patterns/
├── SKILL.md            ← the candidate, generated by this skill
└── .source-blueprint    ← marker file pointing to the blueprint that produced it
```

Where `{topic}` is derived from the blueprint slug by stripping the `-blueprint` suffix and appending `-patterns`. Example: `project-b-pgvector-schema-blueprint.md` → `project-b-pgvector-schema-patterns`. The full slug is preserved; do not truncate semantically.

Generated skills are STAGING. They live under `generated/` until `/skill-validator` approves and `/skill-register` promotes them to `.claude/skills/{topic}-patterns/`.

## Workflow

### Step 1 — Resolve and read the blueprint

1. Resolve `{blueprint-slug}` to `.claude/knowledge-base/discoveries/blueprints/{slug}-blueprint.md`.
2. Verify it exists. If not, exit with error.
3. Verify the blueprint's confidence verdict (read `.claude/knowledge-base/reviews/{slug}-confidence-*.md` if it exists; otherwise re-run `/discover-confidence` quickly to confirm). Abort if verdict is INVALID or NON_SHIPPABLE.
4. Read the full blueprint.

### Step 2 — Extract the load-bearing knowledge

A blueprint contains 4 coverage corners + ADRs + Recommendations + Cross-cutting Comparison + Blocked questions. Not all of those go into the skill.

**Extract (these become the skill body):**

- **ADRs** — each `### Dx` becomes a "Pattern" section in the generated skill. The Decision becomes "Pattern", the Rationale becomes "Why", the Alternatives become "Other approaches considered". Consequences become "When this pattern is the right fit / wrong fit".
- **Recommendations** — each `## Recommendations` entry becomes an "Apply when planning" bullet point in the generated skill, with the linked rule + research question.
- **Cross-cutting Comparison table** — becomes a "Quick reference" table in the skill (collapsed to the most relevant columns).
- **Key citations** — every `.claude/knowledge-base/references/{project}/path:N` that appears in 2+ different blueprint sections is "load-bearing" — include as Evidence link.

**Discard (these are blueprint metadata, not skill knowledge):**

- Halt-loop progress audit trail
- Blocked questions
- Per-iteration progress numbers
- Source plan reference (move to `.source-blueprint` marker file instead)

### Step 3 — Generate the candidate SKILL.md

Use the template at `templates/generated-skill-template.md` and substitute:

- `{TOPIC_SLUG}` — derived from blueprint slug minus the `-blueprint` suffix
- `{SOURCE_BLUEPRINT_PATH}` — full path to the blueprint
- `{PATTERNS}` — extracted from blueprint ADRs (one Pattern section per ADR)
- `{RECOMMENDATIONS}` — extracted from blueprint Recommendations section
- `{APPLIES_WHEN}` — derived from the blueprint's Objective + Recommendations linkage
- `{QUICK_REF_TABLE}` — derived from blueprint Cross-cutting Comparison
- `{KEY_CITATIONS}` — top 5-10 load-bearing .claude/knowledge-base/references/ paths

The generated frontmatter MUST follow Claude Code conventions:

```yaml
---
name: {topic-slug}-patterns
description: <one paragraph synthesizing what this skill encapsulates and WHEN /to-plan should consult it — load-bearing for Step 0 frontmatter scan>
user-invocable: true
allowed-tools: Read Glob Grep
generated-from-blueprint: {blueprint-slug}
generated-at: {YYYY-MM-DD}
---
```

The `generated-from-blueprint` and `generated-at` fields are NOT standard Claude Code frontmatter — Claude Code ignores them. They are audit metadata read by `/skill-validator` and `/skill-register`.

### Step 4 — Write the .source-blueprint marker

A small marker file at `.claude/skills/generated/{topic}-patterns/.source-blueprint`:

```
{blueprint-slug}
{blueprint-confidence-verdict}
{blueprint-confidence-score}
{generation-timestamp-iso}
```

Used by `/skill-validator` to cross-reference back to the blueprint, and by `/skill-register` to audit.

### Step 5 — Recommend next step

Output to stdout:

```
=== Skill-Writer complete ===
Candidate skill: .claude/skills/generated/{topic}-patterns/SKILL.md
Source blueprint: .claude/knowledge-base/discoveries/blueprints/{slug}-blueprint.md
Patterns extracted: N (from N ADRs)
Recommendations consolidated: N

Next: /skill-validator {topic}-patterns
```

## Quality rules (non-negotiable)

1. **Never fabricate a pattern.** If the blueprint has < 1 ADR, the generated skill has < 1 Pattern section — do NOT invent.
2. **Never fabricate citations.** Every `.claude/knowledge-base/references/` path in the generated SKILL.md must come VERBATIM from the source blueprint. The validator will check this.
3. **Description trigger phrases must be specific.** Generated `description` must include 2-3 concrete trigger phrases ("Use when planning pgvector schema", "Use when designing src/local/", etc.). Vague descriptions ("Use for memory stuff") fail validation.
4. **Never overwrite an existing skill.** If `.claude/skills/{topic}-patterns/` (production) OR `.claude/skills/generated/{topic}-patterns/` (staging) already exists, abort with error and report. Manual decision needed: replace, merge, or skip.
5. **No production code patterns.** Generated skills inform `/to-plan`; they do NOT contain executable scripts, hooks, or templates that run. Pure knowledge in markdown.

## Anti-patterns

1. **Generating a skill from a NON_SHIPPABLE blueprint** — propagates defect downstream. Validator will catch but writer must abort earlier.
2. **Single mega-skill from multiple blueprints** — out of scope; each blueprint → one skill. Cross-blueprint synthesis is a different (not yet existing) skill.
3. **Inferring patterns from blueprint prose** — only extract from STRUCTURED sections (ADRs, Recommendations, Cross-cutting Comparison). Free-form prose is not authoritative.
4. **Adding executable functionality** — generated skills are read-only knowledge. Adding `allowed-tools: Bash Write` to a generated skill is forbidden by validator.

## What this skill does NOT do

- Validate the generated skill — that's `/skill-validator`.
- Promote to first-class — that's `/skill-register`.
- Update other skills' SKILL.md to reference the new one — `/to-plan` Step 0 discovers via frontmatter scan, no edits needed.
- Re-generate from same blueprint without manual approval — refuses to overwrite.
