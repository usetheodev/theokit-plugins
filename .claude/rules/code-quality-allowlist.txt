# Code-quality allowlist — explicit exemptions with mandatory sunset.
#
# Authoritative source: rules/code-quality-golden-rule.md § 4 (Allowlist mechanism).
#
# Each ACTIVE entry downgrades a finding by ONE severity level
# (HARD → SOFT_CAP, SOFT_CAP → SOFT_FLOOR). Expired entries (today > SUNSET)
# are IGNORED — the underlying finding re-fires at full severity.
#
# Format (one entry per line):
#   IDENTIFIER | FILE:LINE | SUNSET (YYYY-MM-DD) | RATIONALE
#
# Where:
#   IDENTIFIER  — stable identifier from `code-quality-golden-rule.md § 2`
#                 (e.g., dead_code_unallowlisted_python, symbol_fabrication_typescript)
#   FILE:LINE   — absolute file path within the project + line number where the
#                 finding fires (e.g., src/legacy/foo.py:42)
#   SUNSET      — date in ISO format (YYYY-MM-DD). MUST be ≤ 90 days from entry.
#   RATIONALE   — free-form justification (recommended; not enforced by parser).
#
# Lines starting with `#` and blank lines are skipped.
# Whitespace around `|` is tolerated.
#
# Adding/modifying entries:
#   - REQUIRES CHANGELOG entry under `[Unreleased] § Changed`.
#   - REQUIRES ADR for HARD-level exemptions (`knowledge-base/adrs/`).
#
# A malformed entry triggers the `allowlist_malformed_entry` HARD finding and
# aborts allowlist processing for the current run.
#
# Empty allowlist is the default. Add entries only when the underlying issue
# cannot be fixed within the iteration AND the risk is documented.

# Example (commented):
# dead_code_unallowlisted_python | src/legacy/migrator.py:128 | 2026-09-01 | Legacy migration helper retained for v0.9 rollback path; remove after v1.0 release.
