# AI Gateway (internal) — Roadmap

> Generated by `/roadmap-init` on 2026-06-07 from slug `ai-gateway-vercel-like`.
> This is a **macro** roadmap. Decomposition into tasks is the job of `/to-plan` per milestone.

## Vision

Provide a single internal AI inference entry point — modeled on Vercel AI Gateway's developer experience — that abstracts model providers behind one API, enforces org-wide policy (quota, cost cap, PII redaction), and gives every product team observability without each team rebuilding the same plumbing.

## Problem

**Root problem:** Five product teams have each integrated 2-4 LLM providers directly (OpenAI, Anthropic, Bedrock, vLLM). Quota negotiation, cost tracking, failover, and prompt-injection guardrails are duplicated and inconsistent. When a provider has an incident, every team scrambles independently.

**Whose pain:** Platform team (carries on-call for incidents they did not author), finance (no unified cost attribution), product teams (rebuilding the same retry/fallback every quarter).

## Users

- **Primary:** internal product engineers across 5 product teams (~40 devs).
- **Secondary:** platform on-call team (operational view), finance (cost attribution view).

## Scope

### In scope (V1 — required to call this project alive)

- One HTTP API (`POST /v1/completion`, `POST /v1/embedding`) that fan-outs to any registered provider.
- Provider registry with per-provider auth, rate limit, cost-per-token.
- Per-org and per-project quota enforcement (hard cap, soft warn).
- Failover chain (primary → secondary) on 5xx and rate-limit responses.
- Streaming SSE passthrough.
- Per-request audit log (caller, provider, tokens in/out, latency, cost).

### Explicitly out of scope

- **Building our own inference engine** — *why excluded:* not our business; vendors do it better.
- **Fine-tuning orchestration** — *why excluded:* separate problem with its own lifecycle; revisit after V1.
- **Per-prompt caching** — *why excluded:* requires semantic-equivalence judgement we cannot deliver in V1 cycle.
- **Multi-region active-active** — *why excluded:* single region acceptable for V1; product teams already accept that latency.

> Items in this list are off-limits for V1. To reconsider, write a new roadmap revision — do not silently expand scope.

## Constraints

| Category | Constraint |
|---|---|
| Stack | Python 3.11+ (platform team standard), FastAPI, Postgres for audit log, Redis for quota counters. |
| Compliance / legal | SOC 2 Type II — every prompt/response logged with retention policy. PII redaction at ingress required for marketing-team prompts. |
| Deadline | First product team on-boarded by 2026-09-30 (Q3 OKR). |
| Team | 2 platform engineers + 1 product engineer on rotation; no dedicated SRE. |
| Runtime target | Kubernetes (internal cluster), behind internal mesh. No public exposure. |
| Budget (infra, tooling) | Sub-$2k/mo infra. No managed-gateway purchase (we are the alternative). |

## Success criteria

**V1 ship criterion (measurable):** Two product teams in production using the gateway for ≥80% of their LLM calls for ≥4 consecutive weeks, with P95 latency overhead < 50ms vs direct provider call, and zero provider-credentials leaked outside the gateway.

**North-star metric (tracked post-launch):** Percentage of company-wide LLM API spend routed through the gateway. Target ≥ 90% by 2027-Q2 (proxy for "every team has switched").

## Milestones

### M0 — [ ] Walking skeleton

**Objective:** End-to-end request flow with ONE provider (OpenAI), ONE endpoint (`/v1/completion`), ONE caller (platform team's own dogfood script), deployed to staging cluster.

**Definition of done:**

- [ ] `POST /v1/completion` accepts request, calls OpenAI, returns response.
- [ ] Request and response written to Postgres audit table.
- [ ] Deployed to staging k8s; readiness/liveness probes green.
- [ ] One end-to-end integration test in CI hits real OpenAI (sandbox key).
- [ ] Latency overhead measured and documented (baseline for M5).

**Dependencies:** none.

**Top risks:**

1. OpenAI sandbox quota too low for CI — mitigation: cache one request/response and replay in CI; only nightly hits real API.
2. Streaming SSE not in scope for M0 — explicit: M0 is request/response only.

---

### M1 — [ ] Multi-provider abstraction

**Objective:** Add Anthropic and Bedrock as second/third providers behind the same API contract. Provider chosen by request parameter; no automatic routing yet.

**Definition of done:**

- [ ] Provider interface (`Provider` ABC) with `completion()` / `embedding()` methods.
- [ ] OpenAI, Anthropic, Bedrock implementations passing the same contract tests.
- [ ] Request body specifies `provider` and `model`; gateway dispatches.
- [ ] Audit log records `provider` and `model` per request.
- [ ] Documentation block in repo describing how to add provider #4.

**Dependencies:** M0.

**Top risks:**

1. SDK divergence (OpenAI/Anthropic/Bedrock APIs differ semantically, not just syntactically) — mitigation: limit M1 contract to lowest common denominator; document gaps.
2. Bedrock IAM requires assumed-role chain — mitigation: borrow LiteLLM's bedrock auth pattern (see references catalog).

---

### M2 — [ ] Streaming SSE passthrough

**Objective:** Support streaming for all providers under the same `/v1/completion` endpoint via SSE.

**Definition of done:**

- [ ] `stream: true` in request body triggers SSE response.
- [ ] Provider streams proxied byte-for-byte with normalized event names.
- [ ] Client disconnect cleanly cancels upstream provider stream.
- [ ] Streaming latency overhead < 20ms first-token vs direct provider.

**Dependencies:** M1.

**Top risks:**

1. Disconnect propagation lost across mesh — mitigation: explicit `asyncio.CancelledError` handler.
2. Token counting inaccurate for streaming — mitigation: count on stream-end, not per-chunk.

---

### M3 — [ ] Per-org / per-project quota enforcement

**Objective:** Hard cap and soft warn on token + dollar spend per org and per project.

**Definition of done:**

- [ ] Quota config in Postgres, refreshed every 60s.
- [ ] Redis counters for tokens and dollars per (org, project) per (minute, hour, day).
- [ ] Soft-warn threshold (80%) emits warning header in response.
- [ ] Hard-cap threshold (100%) returns 429 with `X-Quota-Reset` header.
- [ ] Quota state inspectable via internal admin endpoint.

**Dependencies:** M1 (provider abstraction lets us count tokens uniformly).

**Top risks:**

1. Token-counting drift across providers — mitigation: store provider-reported `usage` field; if absent, fall back to tiktoken approximation flagged in audit log.
2. Redis-Postgres consistency window — mitigation: accept eventual consistency; document the 60s drift.

---

### M4 — [ ] Failover chain

**Objective:** On provider 5xx or rate-limit, automatically retry against secondary provider per configured chain.

**Definition of done:**

- [ ] Chain config per (org, model-family) in Postgres.
- [ ] Retry only on 5xx and 429; never on 4xx.
- [ ] Single retry to one fallback (no cascading); failure surfaces to caller with both errors in `X-Failover-Trace` header.
- [ ] Failover events logged separately for alerting.

**Dependencies:** M3.

**Top risks:**

1. Caller already has its own retry → amplified load on fallback. Mitigation: gateway sets `X-Retried-Internally` so caller can choose to suppress its retry.
2. Cost attribution gets fuzzy across providers. Mitigation: audit log records both attempts.

---

### M5 — [ ] Observability surface

**Objective:** Per-request audit + aggregated dashboards (latency, error rate, cost) consumable by on-call and finance.

**Definition of done:**

- [ ] Prometheus metrics: `gateway_request_total{org,project,provider,status}`, `gateway_latency_seconds`, `gateway_tokens_total`, `gateway_cost_dollars_total`.
- [ ] Grafana dashboard committed under `ops/grafana/`.
- [ ] Audit log retention: 90 days hot in Postgres, archived to object storage after.
- [ ] Cost-per-org report exportable as CSV via internal admin endpoint.

**Dependencies:** M0, M3, M4.

**Top risks:**

1. Cardinality explosion on labels — mitigation: cap labels to (org, project, provider, status), no per-user dimension.
2. Audit log table bloat — mitigation: partition by week; archive job in CI.

---

### M6 — [ ] PII redaction at ingress (marketing-team requirement)

**Objective:** Detect and redact PII (email, phone, CPF) in request bodies for orgs flagged `pii_redaction_required=true`.

**Definition of done:**

- [ ] Redactor module called before provider dispatch; pluggable per-org config.
- [ ] Original (unredacted) bodies NEVER persisted to audit log when flag is on; only redacted version stored.
- [ ] False-positive rate measured against fixtures; documented in repo.
- [ ] Compliance review sign-off recorded in `docs/compliance/`.

**Dependencies:** M5 (audit log must already exist to flip its redaction logic).

**Top risks:**

1. Redactor latency adds > 30ms — mitigation: benchmark on M0, choose Presidio or roll-our-own based on perf.
2. False-positive redaction corrupts user prompts — mitigation: allowlist of regex per org for known-safe patterns.

---

### M7 — [ ] First product-team onboarding

**Objective:** First product team migrates ≥80% of LLM calls to the gateway for 4 consecutive weeks.

**Definition of done:**

- [ ] Migration runbook in `docs/runbooks/onboarding.md`.
- [ ] Product team's existing direct integration removed from their codebase.
- [ ] 4-week dashboard snapshot: ≥80% of their LLM volume on gateway, P95 overhead < 50ms.
- [ ] Post-mortem of onboarding written, fed back into runbook.

**Dependencies:** M0-M5 (M6 only if that team has PII requirement).

**Top risks:**

1. Product team feature-blocked on missing capability — mitigation: confirm capability fit before onboarding start, not after.
2. Quota surprises in production — mitigation: shadow-mode period (route through gateway but don't enforce caps) for first week.

---

### M8 — [ ] Second product-team onboarding (V1 ship gate)

**Objective:** Second product team migrates per same criteria — meets the V1 ship criterion.

**Definition of done:**

- [ ] Same DoD as M7 applied to second team.
- [ ] V1 ship criterion declared met in CHANGELOG.md.
- [ ] Retrospective with both onboarded teams; documented in `docs/retros/v1-ship.md`.

**Dependencies:** M7.

**Top risks:**

1. Second team's stack differs enough to expose hidden assumption — mitigation: pick second team with intentionally-different usage pattern (e.g. heavy embedding vs heavy completion).
2. Capacity headroom insufficient for two teams — mitigation: load test on M5 dashboards before M7 starts.
