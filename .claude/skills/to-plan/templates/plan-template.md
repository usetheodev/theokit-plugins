# Plan Template

Every plan generated by `/to-plan` MUST follow this structure. Each section is mandatory unless marked (optional).

This file is the canonical template referenced from `to-plan/SKILL.md § Plan Template`. To produce a new plan, copy everything inside the `<plan-template>` block below into `knowledge-base/plans/{slug}-plan.md` and replace placeholders.

The structure is calibrated against three reference points:

- **RFC tradition** (Rust RFCs, Python PEPs, IETF RFCs) — Motivation, Detailed Design, Drawbacks, Rationale & Alternatives, Prior Art, Unresolved Questions.
- **C4 / ARC42 baseline view** — what exists today before any change is the foundation for evaluating what comes next.
- **ReAct planning** (Yao et al. 2022 — action + reasoning per step) — every atomic step declares both the action and the observable reason.

The goal is **a plan a junior engineer can implement without guessing**. If a junior would have to interpret an instruction, the instruction is incomplete.

# Plan: {Title}

> **Version 1.0** — one-paragraph executive summary explaining what this plan does, why it matters, and what the expected outcome is.

## Goal

**One sentence, action-oriented, measurable.** Mandatory format:

> "Enable {who} to {capability} so that {observable outcome}, measured by {metric}."

Example:

> "Enable `<feature>` users to `<action>` so that `<observable outcome>`, measured by `<test name or metric>` returning `<expected value>`."

**NON-NEGOTIABLE:**
- The Goal MUST fit in one sentence with one named metric. If it cannot, the plan scope is too large — split into multiple plans.
- The primary verb MUST be action-oriented with a clear end-state: `ship`, `enable`, `replace X with Y`, `reduce X from N to M`, `add`, `remove`, `migrate`. Vague verbs are FORBIDDEN as the primary verb: `improve`, `enhance`, `better`, `optimize` (without a number), `clean up`, `refactor` (without a target).
- The metric MUST be observable from outside the implementation: a test that passes, a benchmark threshold, a counter non-zero in integration workload, a file that exists, a coverage percentage. "Code looks cleaner" is NOT a metric.
- The Goal is the contract `/plan-confidence`, `/implement`, and `/review` cite. Changing it after `/edge-case-plan` runs requires bumping the plan version.

## Context

What problem motivates this plan? Cite issues, PRs, ADRs, customer reports, benchmark results, or recent incidents. **No more than 5 paragraphs.** Long forensic detail belongs in the next section.

## Baseline Context (deep review of current state)

> This section is the **"what exists today"** snapshot that lets a junior implementer reason about the change without spelunking the codebase. It is generated from real `git log` + `grep` evidence, not invented. `/plan-confidence` verifies the table cells resolve.

### Files that will be touched

### Current callers / dependents

- **Symbol:** `foo()` in `src/path/to/file.ext`
- **Callers (production):** `src/feature/a.ext:42`, `src/feature/b.ext:18`
- **Callers (tests):** `tests/a.test.ext`, `tests/b.test.ext`

Use `grep -rln 'symbolName' --include='*.<ext>'` to enumerate. Citations MUST resolve.

- **<term-2>** — what it means in this codebase

### Architecture boundaries affected

## Prior Art & Related Work

Sources that informed the design. **Every entry MUST resolve** — `/plan-confidence` checks. At least one of each kind below when applicable.

- **Internal blueprints** (`knowledge-base/discoveries/blueprints/`) — cite `Blueprint §"Q1"` style; section must exist in a real blueprint file.
- **Patterns skills** (`skills/*-patterns/`) — cite the skill name + the specific Pattern or Recommendation consumed. If the plan overrides a registered Pattern, list it in `## ADRs` with rationale.
- **Reference projects** (`knowledge-base/references/{project}/`) — cite `file:line` evidence for any technique borrowed.
- **External literature** (papers, RFCs, docs of libraries you adopt) — link + 1-sentence relevance. Only sources in `rules/discover-web-allowlist.txt` are acceptable for `/discover-execute` allowlisted domains; plans cite freely but the citation must be a stable URL.

If you cannot identify any prior art, write "(none identified — first-of-its-kind in this codebase)" and the edge-case review will challenge that claim.

## Objective

The list of specific, measurable sub-goals that, taken together, compose the **Goal** above. Each sub-goal is a checkbox the implementer can tick off; the Goal is achieved when all sub-goals are checked AND the named metric in the Goal is observed.

- [ ] Sub-goal 1 — specific, measurable
- [ ] Sub-goal 2 — specific, measurable
- [ ] Sub-goal 3 — specific, measurable

If the list of sub-goals has more than ~7 entries, the Goal is likely too broad — split the plan.

## ADRs

Architecture Decision Records for this plan. Each decision gets:
- **ID** (D1, D2, ...) for cross-referencing in tasks
- **Decision** — what was decided
- **Rationale** — why this approach over alternatives
- **Alternatives considered** — at least one rejected alternative, with reason for rejection
- **Consequences** — what this enables and what it constrains

`/plan-confidence` caps the score at 70 if any ADR lacks an alternative.

## Drawbacks & Risks

What does this plan make worse, riskier, or harder to change? Honesty here is the price of admission. **At least 2 entries** — no plan is risk-free.

| Drawback / Risk | Severity | Mitigation | Owner |
|---|---|---|---|
| Migration window leaves users on old schema for ~2h | Medium | Run dual-write for 24h before cutover | DBA |
| New dependency `libX` adds 800KB to bundle | Low | Tree-shake; benchmark before/after | FE |
| ... | ... | ... | ... |

If you cannot identify any drawback, the plan is probably under-scoped OR over-optimistic. `/plan-confidence` caps the score at 70 if this section is missing or has fewer than 2 entries.

## Unresolved Questions

Open questions that this plan cannot answer yet but must be answered before merge. These become MUST-FIX items consumed by `/edge-case-plan`.

- Q1 — what happens if {edge case}?
- Q2 — does {assumption} hold under {load condition}?
- Q3 — ...

Write "(none — every decision is resolved at plan time)" only if you are certain. `/edge-case-plan` will challenge unresolved-question lists that look too short.

#### Deep Dives

#### Pseudo-code / Signatures (optional but recommended for non-trivial tasks)

For tasks whose acceptance criterion is non-trivial, provide a concrete signature + 3–10 lines of pseudo-code + one input/output example. This is the bridge from "the plan said do X" to "I know what X looks like in code".

```pseudocode
function processBatch(items: Item[]): BatchResult
  -- precondition: items.length <= MAX_BATCH (declared by D2)
  for item in items:
    validated = validate(item)
    if validated.error: collect_error
    else: append result
  return { ok: results, errors: errors }

# Example
input:  [{id: 1, amount: 10}, {id: 2, amount: -5}]
output: { ok: [{id: 1, amount: 10}], errors: [{id: 2, reason: "negative amount"}] }
```

Skip for trivially-defined tasks (e.g., "rename a constant"). Anything with a branch / state / loop SHOULD have it.

#### Tasks
Numbered checklist of atomic implementation steps. Each step is one verb + one observable change:
1. Step one — concrete action

3. ...

#### TDD
Strict RED-GREEN-REFACTOR cycle. List every test FIRST:

```
RED:     test_name() — what it asserts (this test MUST fail before implementation)
RED:     test_name_2() — what it asserts
GREEN:   Implement the minimal code to make all RED tests pass
REFACTOR: What cleanup is expected (or "None expected")
VERIFY:  <project test command for this file>
```

#### Concurrency tests (only when applicable)

If this task touches state mutated under concurrency — locks, mutexes, async/await,
goroutines, threading, atomic counters, channels, shared memory, transactions — list
the concurrency tests here. Single-threaded TDD does NOT catch race conditions: the
single-thread execution always interleaves cleanly. A race-aware test is the only
proof that the invariant holds.

Acceptable shapes (pick what fits the stack):

- **Race detector run** — `go test -race ./pkg/...`, `cargo test` under `loom`,
  Python `pytest -p no:cacheprovider` + `pytest-asyncio` with a stress fixture,
  Java JCStress harness, Node `worker_threads` test exercising the shared resource.
- **Atomic-counter invariant** — N concurrent writers each `Add(1)`; assert final
  count == N and no Lost Update.
- **Happens-before observation** — explicit barrier (`sync.WaitGroup`, `Promise.all`,
  `CountDownLatch`) followed by an assert on the observed-after state.
- **Cancellation / timeout** — when concurrency is structured (goroutine + ctx,
  asyncio.Task), assert that cancelling the parent propagates and the child stops.

If the task is genuinely single-threaded (refactor of pure function, UI markup
change, CLI argument parsing), write exactly:

```
(none — single-threaded)
```

`/plan-confidence` only enforces this subsection when the plan's Baseline Context
or Deep Dives contains a concurrency signal (mutex/lock/async/goroutine/atomic/
channel/threading/concurrent). Plans without concurrency are unaffected.

## Failure scenarios (when I/O external)

If the plan touches **non-deterministic external I/O** — HTTP calls, database queries
under transaction, message queues, gRPC, sockets, S3-class object stores, third-party
services — list ≥ 1 failure scenario per external dependency here. Happy-path tests
do NOT prove resilience: production failure modes (timeouts, 5xx, connection reset,
rate-limit, partial writes) are where most outages happen.

For each external dependency, document:

| Dependency | Failure mode | How the test reproduces it | Expected behavior |
|---|---|---|---|
| `payments-api` (HTTP) | 5xx burst (10 consecutive) | mock with `httpx_mock` returning 503 | circuit breaker opens; user sees friendly error; metric `payments_failed` increments |
| `postgres:orders` (DB) | connection reset mid-transaction | testcontainers + `pg_terminate_backend` | retry exactly once; on second failure, surface to user; no partial write to `orders` |
| `nats:events` (queue) | publish ack timeout (3s) | toxiproxy with `latency=5s` | message buffered; retry with backoff; metric `nats_publish_retry` non-zero |

If the plan touches **no external I/O** (pure in-process logic, refactor, UI markup),
write exactly:

```
(none — no external I/O touched)
```

`/plan-confidence` only enforces this section when the plan's Baseline Context, Files
to edit, or Deep Dives contains an external-I/O signal (HTTP client, database driver,
queue, gRPC, socket). Plans without external I/O are unaffected.

### Acceptance Criteria

- [ ] All test suites green (unit + integration + e2e when present)
- [ ] Coverage ≥ 90% on changed files (critical paths: 100%)

- [ ] Zero lint warnings
- [ ] Runtime-metric proof — for any task referencing a counter/metric, that counter is observed non-zero in integration tests

### If Validation Fails

1. Identify which failures are caused by this plan's changes vs pre-existing
2. Fix all plan-caused failures before declaring the plan complete
3. Re-run the validation chain to confirm fixes
4. Pre-existing issues are logged but do NOT block plan completion (document them in the PR description)
